# 커스텀 문제 생성 계획

> 기출을 레퍼런스로 두되, **최신 출제 범위**의 신규 유형과 **기출 변형(숫자·문자·순서 변경)** 모두 커스텀으로 제작합니다.  
> 기술 등록 절차(`traceSteps`, `index.ts`, 검증 스크립트)는 `hello-binary_build_plan.md` §11을 따릅니다. 이 문서는 **무엇을·어떤 순서로·어떤 난이도로** 만들지에 집중합니다.

**작성일:** 2026-07-13  
**근거:** 2026년 1회차 이후 출제 경향 정리 (사용자 제공)  
**갱신:** 기출 변형 · `커스텀-001` · §11 제품 결정 · **문제 뱅크 MD 선작성 → 코드 순차 구현**

## 1. 왜 커스텀인가

| 구분 | 기출 기반 (현재 33문제) | 커스텀 (이번 목표) |
|------|------------------------|-------------------|
| 목적 | 실제 시험 코드 재현·Trace | 유형 **반복 연습** (신규 + 변형) |
| 코드 | 회차 지문과 동일/근접 | ① 신규 설계 또는 ② 기출 골격 + **숫자·문자·순서·이름만 변경** |
| `source` | `"2026년 1회차"` 등 | `"커스텀-001"` (전역 일련번호, 세 자리) |
| 학습 효과 | “이 회차 풀이” (눈에 익으면 쉽게 풀림) | “같은 유형인데 값이 바뀌어도 풀린다” |

기출은 **레퍼런스**로 유지합니다. 커스텀은 두 갈래입니다.

| 갈래 | 설명 | 언제 쓰나 |
|------|------|-----------|
| **신규(New)** | 최신 범위·아직 없는 유형을 처음부터 설계 | 컴프리헨션, 정렬 빈칸 등 갭 |
| **변형(Variant)** | 기존 기출과 **제어 흐름·함정은 동일**, 리터럴·식별자·배열 순서·연산 방향 등만 바꿈 | 연결리스트·상속처럼 이미 많은 유형의 **추가 연습** |

변형이 허용되는 이유: 기출은 몇 번만 풀어도 코드 형태가 눈에 익어 정답이 떠오른다. 숫자·문자·노드 순서만 바꿔도 **다시 한 줄씩 추적해야 하는 새 문제**가 된다.

## 2. 제작 원칙

### 2.1 한 문제 = 한 핵심 함정

- 문제는 **검증하려는 개념 1~2개**만 담는다.
- “포인터 + 구조체 + 비트 + 재귀”처럼 여러 함정을 한 코드에 몰아넣지 않는다.
- Trace가 길어지면(대략 40단계↑) 코드를 쪼개거나 난이도를 `어려움`으로 명시하고 단계를 줄일 방법을 먼저 검토한다.
- **변형**은 원본 기출의 함정을 그대로 가져와도 된다 (함정을 새로 발명할 필요 없음).

### 2.2 시험 톤은 유지 — 신규든 변형이든 “외우기”는 막기

- 지문(`prompt`)은 기존과 같이 시험 문체 유지.
- 정답은 짧고 채점 가능한 문자열(출력값, 빈칸 단어 등)로 유지.
- **신규:** 변수명·구조·값을 자유롭게 설계.
- **변형:** 기출과 뼈대는 같아도 되며, 아래 중 **하나 이상**만 바꾸면 커스텀으로 등록 가능.

### 2.2.1 변형(Variant)에서 바꿔도 되는 것

| 바꿀 것 | 예시 |
|---------|------|
| 숫자 리터럴 | 배열 `{3,1,4}` → `{7,2,9}`, 루프 횟수, 시프트 양 |
| 문자·문자열 | `"ABC"` → `"XKP"`, 출력 구분자 |
| 순서·방향 | 연결리스트 링크 순서, 슬라이스 방향, 순회 방향 |
| 식별자 | 변수·함수·클래스 이름 (학습 방해 없을 정도만) |
| 초기 데이터 배치 | 노드 값 배치, dict 키 집합, 객체 필드 초깃값 |

구조(함정·알고리즘)를 유지한 채 위만 바꿔도 **정답·Trace·explanation은 전부 새로** 작성한다. “기출 답 암기”가 통하지 않으면 충분하다.

### 2.2.2 변형에서만 지키는 최소선

- 기출 파일과 **코드 문자열이 동일**하면 안 된다 (최소 리터럴·순서 변경).
- slug는 기출(`-2025-2` 등)을 쓰지 않고 `-custom-NN` 사용.
- **원본 기출은 학습자 UI·explanation에 노출하지 않는다.** (연상·암기 방지)  
  제작 추적용 원본 slug는 PR·커밋 메시지에만 적어도 된다.

이미 기출이 많은 유형(연결리스트, 상속, 포인터 등)도 **변형 커스텀을 제한하지 않는다.**

### 2.3 Trace가 가르치는 것

커스텀도 Trace·연습장이 핵심이므로, 코드만 어렵게 만들지 말고 **시각화할 포인트**를 미리 고른다.

| 언어 | Trace에서 꼭 보여줄 것 |
|------|----------------------|
| Python | 슬라이스 결과 리스트, 컴프리헨션 중간 집합, dict/set 변화 |
| C언어 | 주소·`*(p++)` 전후 값, 구조체 필드, 비트 결과 |
| Java | 실제 호출된 메서드(동적 바인딩), 생성자 호출 순서, 힙 참조 |
| 공통 | 재귀 호출 스택, 정렬/탐색 루프의 `i`·`j`·swap |

변형은 원본 Trace 골격(줄 흐름·relatedLines 패턴)을 재사용하되, **바뀐 값에 맞게 변수·메모리·정답을 다시 계산**한다.

### 2.4 네이밍·메타 규칙 (커스텀)

| 항목 | 규칙 | 예시 |
|------|------|------|
| **source** (화면 배지) | `커스텀-{NNN}` — **전 언어 공통** 일련번호, 세 자리 zero-pad | `커스텀-001`, `커스텀-002` |
| slug (파일·URL) | `{개념}-custom-{순번}` | `list-comp-filter-custom-01`, `linked-list-relink-custom-01` |
| topic | 원본/§4 주제명 **재사용** 또는 신규 주제 추가 | |
| difficulty | 커스텀은 **`보통` · `어려움`만** (`쉬움` 의도적 미사용) | |
| 기출 slug와 충돌 금지 | 연도-회차 패턴(`-2026-1`)은 기출 전용으로 유지 | |
| 변형·원본 표시 | **학습자 화면·explanation에 원본 미표시** | |

**source 발급 규칙**

- 기출의 `"2025년 1회차"`에 대응하는 커스텀 출처 표기.
- 번호는 Python / C / Java를 **나누지 않고** 사이트 전체에서 `001`부터 순차 증가.
- slug의 `-01`과 source의 `-001`은 **같을 필요 없음** (slug=개념 식별, source=카탈로그 번호).
- **다음 번호:** 등록 전 `src/problems/**`에서 `source: "커스텀-` 최댓값을 확인한 뒤 +1.  
  PR 설명에 `source: 커스텀-0xx`를 적고, (추후) 검증 스크립트로 중복 번호를 검사한다.

등록·검증 체크리스트는 빌드 플랜 §11.3과 동일합니다.  
1단계는 **「신규: 유형 선정 → 코드 설계」** 또는 **「변형: 기출 선정 → 리터럴·순서 변경 → 정답 재계산」**.  
메타 작성 시 `source: "커스텀-00N"`을 반드시 채운다.

## 5. 문제 설계 워크플로

### 5.0 경로 선택

```
신규 ──→ [1N] 유형 ID → 목표·함정 → 코드 초안 → …
변형 ──→ [1V] 원본 기출 slug 선정 → 리터럴·순서·이름 변경 → 정답 재계산 → …
                    ↓
              이후 §11 등록은 동일
```

### 5.0-N 신규

```
[1N] 유형 ID 선택 (예: PY-C)
      ↓
[2N] 학습 목표·함정 한 줄
      ↓
[3N] custom-problem-bank.md 에 code + answer 초안 작성 → 상태 📝
      ↓
[4N] (여러 초안을 모아 두어도 OK)
      ↓
[5N] 하나씩 .ts + traceSteps 구현 → 상태 ✅ · 뱅크 표 갱신
```

### 5.0-V 변형 (권장 — 제작 비용 낮음)

```
[1V] 원본 기출 선정 (뱅크·UI에 원본 미기재)
      ↓
[2V] 리터럴·순서 변경한 code + 재계산 answer 를 뱅크에 📝
      ↓
[3V] .ts 구현 시 원본 Trace 골격 재사용 + 값 전면 교체 → ✅
```

### 5.3 문제 뱅크 워크플로 (확정)

**실제 `.ts` / Trace 전에** [`custom-problem-bank.md`](./custom-problem-bank.md)에 **문제 코드와 정답만** 미리 적는다.

| 단계 | 하는 일 | 상태 |
|------|---------|------|
| 1. 초안 배치 | 뱅크에 code·answer·메타 작성, `커스텀-00N` 예약 | `📝 초안완료` |
| 2. 코드 제작 | `src/problems/{lang}/{slug}.ts` + Trace·index | `🚧` → `✅ 구현완료` |
| 3. 표시 | 뱅크 현황 표·해당 섹션 상태를 `✅`로 변경, 구현 경로 기입 | |

- Trace는 비용이 크므로, **정답이 확정된 초안만** 구현 큐에 올린다.
- 초안은 한꺼번에 여러 개 작성해도 되고, 구현은 **한 문제씩** 완료 표시하며 진행한다.
- §6 백로그의 “가제”는 뱅크에 엔트리로 옮기면서 구체화한다.

### 5.1 난이도 가이드

| 난이도 | 기준 |
|--------|------|
| ~~쉬움~~ | **커스텀에서는 사용하지 않음** (시험 체감에 맞춰 보통 이상만) |
| 보통 | 개념 1~2개, 출력 추적에 집중, 대략 15~25단계 |
| 어려움 | 혼동 포인트 명시적(예: `*p++`), 또는 재귀/다중 객체, 25단계↑ |

기출에 남아 있는 `쉬움`은 그대로 두되, **신규·변형 커스텀에는 `쉬움`을 넣지 않는다.**

### 5.2 정답 형식

- **출력 추적형** (기본): `print`/`printf`/`System.out` 최종 결과.
- **빈칸형** (AL-B): `answer`에 채울 코드 조각 **하나**로 고정.  
  - 채점은 `normalizeAnswer`(개행 통일 + trim)만 수행 → **공백·대소문자·동의어 복수 정답은 허용하지 않음**.  
  - 지문/`output`에 “정확히 아래 형식으로 작성”을 명시하고, explanation에 허용 표기를 한 가지로 못 박는다.
- **키워드형** (JV-B): `super` / `this` 등 — 기존 기출과 같이 **단일 문자열**.


**유형이 이미 많아도 변형 커스텀은 환영**한다 — 제약은 “기출 code와 완전 동일만 금지” 수준으로 둔다.
